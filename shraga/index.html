The first ten minutes

Today, every Sentry alert lands in a Slack channel. Whoever's around picks it up and burns ten minutes on mechanical work before any real thinking starts.

What an engineer does

  1. Open Sentry. Read the stack trace.
  2. Jump to Coralogix. Filter by service, time, level.
  3. Open GitHub. Blame the suspect line.
  4. Check recent commits in the file.
  5. Write findings into a Slack thread.

What it costs

Six tabs, ~10 minutes per incident, before a single product question gets answered.

Multiply by every alert, every night, every engineer. The expensive part isn't the fix — it's the warm-up.

What Shraga is

An autonomous incident commander. Watches the alerts, runs the warm-up, hands you back an issue that's already half-solved.

The investigation loop

Six phases. The agent commits to a plain-English story before it acts.

  1. 01 Triage Read the alert. Compute the dedup label. Check if Shraga already filed an issue.
  2. 02 Investigate Pull Sentry events, Coralogix logs, recent commits, blame suspect lines.
  3. 03 Probe Verify or falsify each suspect against reality (probe tools wiring in next phase).
  4. 04 Hypothesize Commit to a single root-cause story in plain English. State confidence as a percentage.
  5. 05 Draft fix Sketch the smallest possible fix. No writes. Decide whether to delegate to Claude Code.
  6. 06 File New issue OR comment on the existing one. Never opens a PR directly.

Phase 4 is the agentic-discipline beat. Forcing a plain-English root cause with a numeric confidence — before any write — is what separates investigation from hallucination.

The discipline

Two design choices that make Shraga more than a wrapper.

1. The PR/Issue gate — the agent knows what it doesn't know

High confidence · low risk

≤ 3 files · ≤ 30 lines · no protected paths · ≥ 99% confidence · no new deps · no renamed exports
→ files issue tagged @claude /remote-issue-implementation. Claude Code picks it up and opens the PR. A human still merges.

Anything else

→ plain issue with a ## Suggested approach section. Humans take it from there. Same artifact shape, no auto-tag.

2. Dedup as a feature, not a guard

Every trigger produces a deterministic dedup key — e.g. shraga-sentry-6937875213. Before opening anything, Shraga looks up that label on GitHub.

A repeat Sentry firing comments on the existing issue with new evidence — new event count, new timestamps, new context — instead of spawning a duplicate.

Wrappers spawn duplicates. Shraga accrues evidence.

What 10× actually means here

Three measurable axes, not vibes.

Time

~10 min of mechanical engineer warm-up
→ ~90 s of agent investigation

Stack trace fetched, logs queried, suspect lines blamed, evidence linked — autonomously.

Cognitive load

6 tabs — Sentry, Coralogix, GitHub blame, GitHub commits, Slack thread, codebase
→ 1 GitHub issue with linked evidence + a concrete suggested fix

Coverage

Humans skim only the loudest alerts. Shraga triages 100% — and dedup means the noise doesn't grow.

Composition

Shraga → Claude Code → human. Two agents in a pipeline, with the human as the final approver. Same shape generalizes to any on-call task that's currently a Slack thread.
