Two callers

One tool surface, two kinds of caller.

You, vibe coding

Your Claude / Cursor reads live AI Tutor data as the logged-in user. No manual SQL, no curl, no copy-pasting Mongo IDs.

  • "Show me the lifecycle of student X."
  • "Look up user Y: cohort, status, last activity."
  • "Block user Z, here's the reason."

In-app agents

Server-side agents (e.g. the Tutor) act on behalf of a user with a scoped token the user themselves cannot mint.

  • Tutor decides a lesson is complete → calls the tool.
  • Student cannot call the same tool to self-complete.
  • Same surface, different visible tools per actor.

MCP in 60 seconds

JSON-RPC over HTTP. One endpoint, two methods you'll touch.

  • Endpoint: POST /api/mcp/user on mcp.maestro.org
  • tools/list: agent asks "what can I do here?"
  • tools/call: agent invokes a tool with validated input
  • Bearer auth: OAuth 2.1 bearer token (we use a JWT) in the Authorization header

Built on the official @modelcontextprotocol/sdk. Works with Claude Code, the Anthropic API MCP connector, the OpenAI Responses API MCP tool, and any compliant client.

What's there today

Tools split by whose data they touch.

Self-acting (tools/self/)

Operate on the caller only, never take a target id. Per-tool actor gating decides who actually sees them.

  • get_profile (any actor)
  • mark_todo_actions_viewed (any actor)
  • send_email: agent only (real email; humans don't get this)

Admin (tools/admin/)

Take a target userId / email. Gated to human admins with the right permission.

  • get_user
  • block_user / unblock_user
  • get_student_lifecycle

Folder location is a code-organization convention. Visibility is decided by the @McpActorAccess and @McpAccess decorators on each method.

The token

Every call carries a signed JWT. Five claims do all the work.

A JWT is a small signed envelope: three base64 chunks — header, claims, signature — joined by dots (xxx.yyy.zzz). The server checks the signature, then trusts the claims inside. The five rows below are those claims. Paste one into jwt.io to see the parts.

Claim Value What it means
sub user id Whose data the call sees.
actor 'user' | 'agent' Who is holding the token. Routes which tools are visible.
scope mcp-user Which surface this token unlocks.
iss ai-tutor Issuer.
aud mcp-user Audience.

McpUserGuard verifies the JWT, loads the user, resolves permissions, and attaches { user, actor } as McpContext.

Two ways to get a token

Flow A: OIDC

Human in a client (Claude Code, Cursor). Browser-based consent. The token lands in the client automatically.

Result: token with actor='user'.

Flow B: Backend-minted agent token

The backend mints a scoped token so one of its own agents can act on a user's behalf. No browser, no consent click. The trigger comes from inside the system.

Two shapes:

  • Live: a user action spawns an agent (tutor session starts when a student opens a lesson).
  • Scheduled: a job iterates a cohort and spawns a per-user agent (re-engagement emailer, weekly summary, onboarding nudge).

Result: token with actor='agent', only the backend can produce it.

OIDC, briefly

OAuth 2.0 + an identity layer. Standard vocabulary so any compliant MCP client can connect itself.

  • Authorization Server: issues tokens. Lives on mcp.maestro.org.
  • Resource Server: the MCP surface that accepts tokens.
  • Dynamic Client Registration (DCR): the client (Claude Code) registers itself on the fly. No human ops step.
  • Discovery: the client fetches a well-known URL and learns where to send the user, where to refresh tokens, what scopes exist.
  • Authorization Code + PKCE: the grant type. Browser-mediated consent, no client secret in the wild.
  • Identity provider: Google, hard-gated to @masterschool.com accounts. Anyone else hitting the consent screen is rejected before a token is ever issued.

Net effect: paste the MCP URL into Claude, sign in with your Masterschool Google account, done.

Built on

  • oidc-provider (panva): the certified Node OIDC implementation. We get DCR (RFC 7591), PKCE, Resource Indicators (RFC 8707), discovery, and JWKS rotation out of the box. Writing this ourselves means shipping crypto and spec compliance, which is a bad time. Author is on the OAuth WG; the library is what most production Node OIDC servers run.
  • passport-google-oauth20: handles the Google sign-in round-trip on the back of the OIDC interaction screen. The standard choice in Node, boring, exactly what we need.

Flow A: OIDC, end to end

What actually happens between pasting the URL and the first tool call. The whole dance is automatic; the user just clicks "Sign in with Google."

  1. Paste the MCP URL into Claude Code (or any compliant client). The client tries POST /api/mcp/user with no token.
  2. MCP responds 401 with WWW-Authenticate: Bearer realm="mcp-user". That header is the cue: there's an auth server somewhere; go find it.
  3. Claude fetches /.well-known/oauth-protected-resource (RFC 9728) on the resource host. The response says: "the authorization server for me lives at mcp.maestro.org/api/oidc."
  4. Claude fetches /.well-known/oauth-authorization-server (and/or openid-configuration) at the AS. That metadata lists the registration, authorization, and token endpoints, the JWKS URI, supported scopes, and that PKCE is required.
  5. Dynamic Client Registration (RFC 7591): Claude POSTs its name, redirect URI, and grant types to the registration endpoint. Gets back a client_id. No client secret (it's a public client).
  6. Claude generates a PKCE pair: a random code_verifier and its hash code_challenge = SHA256(verifier). Keeps the verifier in memory.
  7. Browser opens to the authorize endpoint with the client_id, redirect URI, scopes, and the code_challenge.
  8. mcp.maestro.org renders /api/oidc/login: "Sign in with Google."
  9. Google authenticates the user and returns the identity. The OIDC server checks the email domain; anything that doesn't end in @masterschool.com is rejected before a token is ever issued.
  10. Consent screen shows the client name and redirect URI (so the admin can spot a phishing client registered via open DCR). For CLI and desktop clients (Claude Code, Cursor) the redirect URI will be a http://127.0.0.1:<port> loopback address; that's normal. User clicks Authorize.
  11. AS redirects back to Claude with a one-time authorization code. Claude POSTs the code plus the original code_verifier to the token endpoint. AS checks SHA256(verifier) == challenge; if matched, issues an access token (actor='user') and a refresh token.
  12. Claude retries POST /api/mcp/user with Authorization: Bearer .... MCP returns the tools/call result, filtered by role + actor.
  1. Steps 1–4 Discover 401 challenge, then two .well‑known lookups
  2. Step 5 Register Dynamic Client Registration, gets client_id
  3. Steps 6–10 Authorize PKCE pair, Google sign-in, domain check, consent
  4. Step 11 Exchange Code + verifier → access & refresh tokens
  5. Step 12 Call Bearer token on POST /api/mcp/user

What is PKCE?

Proof Key for Code Exchange (RFC 7636). It defends the authorization code from interception in public clients (CLIs, desktop apps, browser extensions) that can't safely store a client secret.

The mechanic in one sentence: the client commits to a secret before the redirect (sends the hash), then proves it owns that secret after the redirect (sends the original) when exchanging the code. An attacker who intercepts the auth code on the redirect URL has nothing to do with it; without the verifier, the token endpoint refuses the exchange.

Read the full spec

  • MCP Authorization (the protocol that wires all of this together): modelcontextprotocol.io/specification
  • RFC 9728 · OAuth 2.0 Protected Resource Metadata: datatracker.ietf.org/doc/html/rfc9728
  • RFC 7591 · Dynamic Client Registration: datatracker.ietf.org/doc/html/rfc7591
  • RFC 7636 · PKCE: datatracker.ietf.org/doc/html/rfc7636

Flow B: Backend mints for an agent

The same call surface, but the trigger comes from inside the system.

  1. Trigger fires. Either a user action (student opens a lesson, tutor session spawns) or a scheduled job (cron iterates a cohort, one agent per student).
  2. Backend mints an MCP token: sub=userId, actor='agent'. No browser, no consent.
  3. Backend spawns the server-side agent with the token.
  4. Agent calls tools/list with the bearer header.
  5. MCP returns only the tools gated to actor='agent'. The student's own user-actor token would never see them.
  6. Agent calls tools/call complete_lesson (or send_reengagement_email, etc.).
  7. MCP returns the result.

Actor-gating is enforced twice: tools the agent shouldn't see are filtered out of tools/list, and direct tools/call attempts on hidden tools are rejected.

When to use which

Flow A: OIDC Flow B: Agent mint
Caller A human, in an MCP client A server-side agent
Consent Browser, every new client Implicit; trigger originates inside the system
actor claim 'user' 'agent'
Who can mint The OIDC server, after user consent The backend, for its own agents only
Use case Vibe coding, admin debugging Tutor acting on the student's behalf; scheduled jobs running a per-user agent (re-engagement, summaries)

Identity is not authorization

The token says who you are. The decorators decide what you can call. The handler runs the action.

  1. Layer A: the actor claim

    Routing hint: human vs autonomous agent. Picks which tools show up in tools/list. Not a security boundary on its own; the decorators below are.

  2. Layer B: decorators (the auth boundary)

    @McpActorAccess(['user'])
@McpAccess({ roles: [Role.SuperAdmin] })
@McpAccess({ permissions: ['users:block'] })
@McpPolicies(TargetUserInCallerCohortPolicy)

    This is where authorization lives. Two flavors stack here:

    • Static, on the caller's claims. @McpActorAccess and @McpAccess filter tools/list and reject tools/call based on actor, roles, and permissions. Cheap, evaluated up front.
    • Dynamic, on the validated input. @McpPolicies runs after Zod parses the input, before the handler runs. Use it when authorization depends on the arguments (e.g. "this target user belongs to the caller's cohort"). First throw wins; the tool body never runs.

    Stack to AND. No decorators = visible to any authenticated caller. If a tool is sensitive, the decorator stack on the method is the only thing standing between the caller and the handler.

  3. Layer C: the handler runs your action

    Once a call passes the decorators, the CQRS handler executes the domain action: state-machine checks, idempotency, side effects, persistence. It enforces invariants ("can't complete an unstarted lesson"), not caller identity. If you need defense-in-depth on a sensitive tool, the handler can re-check; by default it trusts that authorization happened upstream.

    Decorators gate. Handlers run.

Where it runs

Same image, two services, two hostnames.

  • api.maestro.org: main API task. MCP_ENABLED unset. MCP routes return 404.
  • mcp.maestro.org: dedicated ECS service running the same image with MCP_ENABLED=true and start:mcp:prod.

Why the split: isolate MCP load and blast radius from the main API, and keep the OAuth/OIDC routes off the main hostname.

Adding a tool

The vibe-coding slide. Use the add-mcp-tool skill; it walks the decision tree.

The 80% case

  1. Pick self/<concern>/ or admin/<concern>/.
  2. Write a Zod input schema. Every field must .describe(...). Agents only see what tools/list returns.
  3. Write a thin tool method: read ctx, dispatch one CQRS query/command, return the result.
  4. Stack decorators if access needs narrowing. Add tests under backend/test/mcp/user/tools/....

What a tool actually looks like

@McpTool({
  name: 'get_profile',
  description:
    'Return the authenticated user profile. Use to read identity, contact, or program info for the current user.',
  inputSchema: getProfileInputSchema,
})
getProfile(_input: GetProfileInput, ctx: McpContext) {
  return this.queryBus
    .execute(new GetUserQuery(ctx.user._id.toString()))
    .then((user) => {
      if (!user) throw new McpToolError('user not found');
      return user;
    });
}

The four decorators

Decorator When evaluated What it does
@McpTool({ name, description, inputSchema }) Tool registration (boot) Declares the tool contract. Required. Without it, the method is just a class method, not an MCP tool.
@McpActorAccess([...]) Discovery + dispatch Filters by ctx.actor. Any-of match. A tool with ['user'] is invisible to agent tokens (and vice versa). Omit for "any authenticated caller."
@McpAccess({ roles, permissions }) Discovery + dispatch Filters by the user's app role and permissions. Roles: any-of (one match passes). Permissions: all-of (every title required). When both are given, both must hold (AND across dimensions).
@McpPolicies(...PolicyClasses) Per call, after input validation Runs domain policies against the validated input. First throw wins; the tool body never runs. Use for input-shaped checks that need DI (e.g. "this email actually belongs to a student in the caller's cohort").

Stack them; they AND together. @McpTool is required, the other three are optional and additive. @McpActorAccess + @McpAccess filter visibility; @McpPolicies validates input; the domain handler is the real boundary.

Three rules

  • Tools are thin and CQRS-agnostic. Business logic lives in handlers.
  • Context stays inside the framework. Don't thread McpContext into a domain handler. Pull values out and pass them explicitly.
  • Describe every field. The discovery test fails if you don't.
